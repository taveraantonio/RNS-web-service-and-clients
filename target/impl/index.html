RNS Service

Implementation Choices

The implementation made on the RNS Service followed the design choices explained here

First of all, the service is made always avaliable; it never terminates even when errors are detected, but throws different exception according to what happen.

The service does not provide data persistency (data are stored in main memory) but manage concurrency. All the idempotent methods are managed to not create undesired situation. Multiple clients can operate concurrently on the same server without restriction. More specific, Java thread-safe collection are used like the ConcurrentHashMap.
Three main classes are created for the service, the first is the Resource class (the part that takes care about interfacing with network), then the Service class (that is the implementation of the logic) and final the Data Storage class (where data are stored). Shared data are avoided in resource and application logic class and demanded only to the Data Storage in order to satisfy concurrency principles.

HATEOAS is respected; clients can get all the possible hyperlinks requesting, through the GET method, the RnsRoot.

Query parameters are also implemented in the system to let user to decide what to get (they are made avaliable in some GET request, for example the getVehicles, the getRoadSegment or the getPlaces).

The service support both XML and JSON mediatype for all the operations, both for the incoming and the outcoming requests.
XML Validation and JSON Interceptor are implemented to validate all the request (incoming and outcoming) against the schema.

Information hiding principles are followed too; the client see only what he needs and nothing about implementation or something else of the service.

The part that is omitted, but described in the design phase, is the handle of the paging mechanism. It is implemented but not working yet. It can be made avaliable in a future update of the system.
