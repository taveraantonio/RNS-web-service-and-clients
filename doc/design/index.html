RNS Service

Design Explanation

The design of the RNS Service is based on a constrained design approach with the aim to follow as much as possible the REST principles.

The service offered varies according to the user; it can be an administrator user or a vehicle user. The service is base on multiple interface:

  • rns/vehicles inteface: that allows access to all the information about vehicles, allows insertion, modification and deletion of vehicles
  • rns/places interfaces: that allows access to all the information about places
  • rns/connections interfaces: that allows access to all the information about connections

The service is organized in order to respect information hiding principles; there is a clear separation between interface and implementation; clients see only what is needed.

The service limit the number of interaction that are necessary to access the server in order to minimize coupling.

Service is also designed to know how to react to different cases that might occur. It is designed to react to all the possible exceptions that could happen. This is in order to make the service more robust and secure.

The majority of method described are idempotent method and the size of message is also limited; the service is also prepared to handle paging mechanism in the near future.

As said above, service is designed to follow REST principles. The APIs are easy to understand and self describing: plural names are used to describe collections while singular names are used to represent single entity. Resurces are represented in a hierarchical way and URLs reproduce the hierarchical relationship of resources. HATEOAS principles are followed too.
